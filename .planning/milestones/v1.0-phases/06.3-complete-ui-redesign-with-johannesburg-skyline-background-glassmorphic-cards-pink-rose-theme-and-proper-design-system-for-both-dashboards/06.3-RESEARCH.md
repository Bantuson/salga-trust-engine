# Phase 06.3: Complete UI Redesign - Research

**Researched:** 2026-02-12
**Domain:** Visual identity redesign, image optimization, CSS theming, glassmorphism
**Confidence:** HIGH

## Summary

Phase 06.3 transforms the existing dark navy UI (#0A0E1A) into a pink/rose (#cd5e81) dominant brand identity with the Johannesburg skyline as hero imagery. This is a **re-skin operation**, not a restructure—all existing page layouts, components, and functionality remain unchanged. The research confirms that the required technologies (CSS custom properties, GSAP ScrollTrigger, glassmorphism via backdrop-filter) are production-ready and well-supported.

**Critical architectural insight:** The existing codebase already has all necessary infrastructure in place. Both dashboards share `design-tokens.css` and `animations.css`, use GSAP for animations, have glassmorphic components (GlassCard), and support responsive design. This phase primarily involves CSS token updates, asset management, and selective component removal (3D globe).

**Primary recommendation:** Execute this as a systematic CSS token migration rather than a ground-up redesign. Update `design-tokens.css` first, then propagate changes through both apps in lockstep to maintain visual parity.

### Pattern 2: Skyline Image Integration with Scroll Fade

**What:** Hero background that fades to solid color on scroll using GSAP ScrollTrigger

**When to use:** Landing page hero, login page, request access page

**Example:**
```tsx
// frontend-public/src/pages/LandingPage.tsx

import skylineImage from '../assets/skyline.png';

export function LandingPage() {
  const skylineRef = useRef<HTMLDivElement>(null);

  useGSAP(() => {
    if (skylineRef.current) {
      gsap.to(skylineRef.current, {
        opacity: 0,
        ease: 'none',
        scrollTrigger: {
          trigger: skylineRef.current,
          start: 'top top',
          end: 'bottom top',
          scrub: true,
        },
      });
    }
  }, { scope: skylineRef });

  return (
    <main className="landing-page">
      <section className="hero-section" style={{ position: 'relative' }}>
        {/* Skyline background layer */}
        <div
          ref={skylineRef}
          className="skyline-background"
          style={{
            position: 'absolute',
            inset: 0,
            backgroundImage: `url(${skylineImage})`,
            backgroundSize: 'cover',
            backgroundPosition: 'center',
            zIndex: 0,
          }}
        />

        {/* Pink background layer (revealed as skyline fades) */}
        <div
          style={{
            position: 'absolute',
            inset: 0,
            background: 'var(--surface-base)', // #cd5e81
            zIndex: -1,
          }}
        />

        {/* Hero content */}
        <div className="hero-content">
          <h1>Transparent Municipal Services</h1>
        </div>
      </section>

      {/* Remaining sections scroll over solid pink */}
      <ProblemSection />
      <SolutionSection />
      {/* ... */}
    </main>
  );
}
```

**Source:** [GSAP ScrollTrigger community examples](https://gsap.com/community/forums/topic/40938-fade-background-images-in-out-while-scrolling-different-sections/)

**What:** Differentiate card prominence using mixed tint and blur levels

**When to use:** All card-based UI elements across both dashboards


**Assignment recommendations (Claude's discretion):**
- **default** (white frost, subtle blur): Metric cards, stat displays, content cards
- **elevated** (pink frost, medium blur): Modal overlays, feature highlights, dashboard panels
- **interactive** (white frost, subtle blur): Buttons in card form, navigation cards, clickable tiles

### Anti-Patterns to Avoid

- **Don't hardcode colors in components:** Always use CSS custom properties, never `#cd5e81` directly
- **Don't animate backdrop-filter:** Extremely expensive — keep blur static, animate opacity/transform only
- **Don't use heavy blur (32px+):** Exponentially degrades performance, especially on mobile
- **Don't set skyline on all pages:** Per user decision, only hero/login/request-access get skyline
- **Don't keep 3D globe:** User explicitly decided to remove it entirely — delete Globe3D imports

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Image optimization | Custom compression scripts | `vite-plugin-image-optimizer` with sharp | Handles WebP/AVIF conversion, responsive srcsets, build-time compression |
| Scroll-driven animations | Custom Intersection Observer logic | GSAP ScrollTrigger (already installed) | Smoother scrubbing, better performance, less code |
| Color palette generation | Manual color picking | HSL-based token system | Mathematically linked variations, easier maintenance |
| Glassmorphic blur fallbacks | Canvas blur polyfills | Accept 95% support, skip fallback | Diminishing returns — ~5% unsupported browsers get solid backgrounds gracefully |

**Key insight:** Modern browser support for backdrop-filter (~95% as of 2025) means fallback complexity isn't justified. Unsupported browsers automatically ignore the property and render solid backgrounds, which is acceptable degradation.

**Source:** [Glassmorphism Design Best Practices 2026](https://invernessdesignstudio.com/glassmorphism-what-it-is-and-how-to-use-it-in-2026)

## Common Pitfalls

### Pitfall 1: Insufficient Color Contrast on Pink Background

**What goes wrong:** Light text on pink background (#cd5e81) can fail WCAG AA contrast requirements

**Why it happens:** Pink is mid-luminosity (~50%), making both white and dark text challenging

**How to avoid:**
- Test all text colors with contrast checker (minimum 4.5:1 for body text, 3:1 for large text)
- Use `--text-primary: #f9f7f8` (very light, not pure white) for maximum contrast
- Add subtle text shadows for readability on busy skyline: `text-shadow: 0 1px 3px rgba(0,0,0,0.3)`

**Warning signs:**
- Headings look washed out or hard to read
- User squinting during user testing
- Automated accessibility audits flagging contrast issues

**Source:** [Glassmorphism Accessibility Best Practices](https://uxpilot.ai/blogs/glassmorphism-ui)

### Pitfall 2: Skyline Image Performance on Mobile

**What goes wrong:** Full-resolution skyline PNG (likely 1920x1080+) causes slow load times on mobile, janky scroll

**Why it happens:** Single large image served to all devices, no responsive optimization

**How to avoid:**
- Add `vite-plugin-image-optimizer` to compress PNG and generate WebP/AVIF variants
- Use `<picture>` element or CSS `image-set()` for responsive delivery:
  ```css
  .skyline-background {
    background-image: image-set(
      url('/assets/skyline-mobile.webp') 1x,
      url('/assets/skyline-desktop.webp') 2x
    );
  }
  ```
- Consider lower resolution for mobile (800px wide vs 1920px desktop)
- Lazy-load skyline on pages below the fold (not hero, but login/request-access pages)

**Warning signs:**
- Lighthouse performance score drops below 80
- First Contentful Paint (FCP) > 2.5s on 3G
- Scroll lag during GSAP fade animation

**Source:** [React Image Optimization Techniques](https://uploadcare.com/blog/react-image-optimization-techniques/)

### Pitfall 3: Glassmorphic Blur Performance on Lower-End Devices

**What goes wrong:** Multiple `backdrop-filter: blur()` elements cause frame drops on integrated GPUs

**Why it happens:** Blur is GPU-intensive, especially when stacked (multiple overlapping glass cards)

**How to avoid:**
- Limit to 2-3 glassmorphic elements per viewport
- Use subtle blur (8-12px) by default, reserve medium blur (16-24px) for primary focus elements only
- Add `will-change: backdrop-filter` only on elements that animate (not static cards)
- Test on integrated GPU devices (e.g., older MacBook Air, mid-range Android)
- Provide reduced-motion fallback: static tinted backgrounds, no blur

**Warning signs:**
- Frame rate drops below 30fps on card hover
- Browser DevTools Performance tab shows long GPU tasks
- Fans spin up on laptop during interaction

**Source:** [Glassmorphism Performance Optimization](https://playground.halfaccessible.com/blog/glassmorphism-design-trend-implementation-guide)

### Pitfall 4: Forgetting to Remove Globe Imports

**What goes wrong:** Dead code remains in bundles, users see console errors from lazy-loaded Globe3D, bundle size bloat

**Why it happens:** User decision to remove 3D globe requires manual deletion of imports, not just commenting out

**How to avoid:**
- Search for all `Globe3D` imports: `grep -r "import.*Globe3D" frontend-public/ frontend-dashboard/`
- Delete files: `Globe3D.tsx`, `Globe3DSmall.tsx`, `GlobeLoadingSkeleton.tsx`
- Remove React Three Fiber dependencies if no longer used: `@react-three/fiber`, `@react-three/drei`, `three`
- Verify no broken imports with TypeScript build: `npm run build` in both dashboards

**Warning signs:**
- TypeScript errors after removing globe components
- Lazy-loaded suspense fallbacks hanging indefinitely
- Bundle size doesn't decrease after removal

**Source:** [React Three Fiber Cleanup Best Practices](https://github.com/pmndrs/react-three-fiber/discussions/564)

### Pitfall 5: Inconsistent Theme Between Dashboards

**What goes wrong:** Public and municipal dashboards drift apart visually, undermining "identical theme" requirement

**Why it happens:** Developers update one `App.css` but forget to sync the other

**How to avoid:**
- Update `shared/design-tokens.css` FIRST, then both dashboards inherit automatically
- After token updates, run side-by-side visual regression tests (manual or automated screenshots)
- Document "no dashboard-specific overrides" rule — all colors MUST come from tokens
- Use Git pre-commit hook to check for hardcoded colors (reject commits with `#cd5e81` in component files)

**Warning signs:**
- One dashboard shows navy remnants while the other is pink
- Text colors differ between login pages
- GlassCard variants look different across apps

## Open Questions

### 1. Exact Pink Border Implementation

**What we know:** User wants pink border from reference image as literal UI element, not just mockup decoration

**What's unclear:** Should this be:
- A border on the skyline background div itself (`border: 20px solid #cd5e81`)?
- A separate overlay frame element positioned absolutely?
- Part of the skyline image asset (baked into PNG)?

**Recommendation:** Implement as separate overlay div for flexibility:
```css
.skyline-frame {
  position: absolute;
  inset: 0;
  border: 20px solid rgba(205, 94, 129, 0.4);
  pointer-events: none;
  z-index: 2;
}
```
This allows responsive sizing, animation if needed, and doesn't require re-exporting the image.

### 2. Recharts Color Palette for Pink Theme

**What we know:** Chart colors need adaptation for pink background readability

**What's unclear:** Which existing chart colors can stay vs which need replacement?

**Recommendation:** Keep teal/gold as accents, replace coral primary:
```css
--chart-primary: #ffd54f;       /* Gold (good contrast on pink) */
--chart-secondary: #00bfa5;     /* Teal (shifted for pink) */
--chart-tertiary: #7c4dff;      /* Purple (better than original violet) */
--chart-quaternary: #ff6e40;    /* Orange (distinct from background) */
```

Test with actual dashboard charts to verify WCAG AA contrast.

### 3. Loading Skeleton Colors

**What we know:** Current skeletons use rgba(255, 255, 255, 0.05) — designed for navy background

**What's unclear:** Best approach for pink background — lighter or darker?

**Recommendation:** Use darker pink tones for skeleton shimmer:
```css
--skeleton-base: rgba(163, 72, 102, 0.3);  /* Darker pink */
--skeleton-highlight: rgba(205, 94, 129, 0.5);  /* Primary pink */
```

This creates contrast while staying within brand palette.
