# Phase 34.1: Demo Mode Fetch Fixes for IDP and SDBIP Pages - Research

**Researched:** 2026-03-03
**Domain:** React frontend — demo mode guards, secondary fetch patterns, loading state management
**Confidence:** HIGH

## Summary

Phase 34.1 is a targeted bug-fix phase addressing a very specific and well-understood problem: the `DEMO_MODE` guard is correctly placed on the primary (initial) fetch callbacks of the IDP and SDBIP pages, but is absent from secondary fetches that are triggered by user interaction (clicking an IDP cycle card navigates to `/pms/idp/:cycleId` — the `IdpDetailPage` — and clicking an SDBIP scorecard card navigates to `/pms/sdbip/:scorecardId/kpis` — the `SdbipKpiPage`). There are also sub-fetches on already-fixed pages like `GoldenThreadPage.fetchThread()` that could hit the API when `selectedCycleId` changes.

The existing `DEMO_MODE` constant (`src/lib/demoMode.ts`) is a simple compile-time boolean derived from `VITE_DEMO_MODE`. The fix pattern is already established in the codebase: import `DEMO_MODE`, guard each `useCallback` fetch function at the very top with `if (DEMO_MODE) { setData(DEMO_DATA); setLoading(false); return; }`. Every affected page already has a `DEMO_*` constant defined. The only missing piece is applying this guard to every fetch callback on the secondary pages.

The root cause of the "UI glitch" is that secondary pages attempt real `fetch()` calls against `/api/v1/...` which fail (since no backend is running in demo mode), causing `loading` state to briefly enter a loading spinner, then fall into the error branch or the fallback branch, creating a visible flash. Because demo data constants already exist for both `IdpDetailPage` and `SdbipKpiPage` (via the parent pages' `DEMO_CYCLES` / `DEMO_SCORECARDS` / `DEMO_KPIS`), the fix is purely additive — no new data structures need to be invented.

**Primary recommendation:** Add `DEMO_MODE` guards to `fetchAll` in `IdpDetailPage`, to `fetchAll` and `handleTransition`/`handleAddGoal`/`handleAddObjective` mutations in `IdpDetailPage`, and to `fetchAll` in `ActualsPage` and `EvidencePage` if those pages are also accessed during demo. The visual stability fix for `SdbipKpiPage` involves ensuring `setLoading(false)` is called synchronously when `DEMO_MODE` is true before any React paint cycle.

### Pattern 1: Demo Mode Guard in fetchCallback (Established Pattern)

**What:** Check `DEMO_MODE` at the very top of a `useCallback` fetch function before any `setLoading(true)` or API call. Set demo data and return immediately.

**When to use:** Every `useCallback` that makes `fetch()` calls and is invoked in a `useEffect` on mount.

**Example (from `SdbipKpiPage.tsx` — correct implementation):**
```typescript
// Source: frontend-dashboard/src/pages/SdbipKpiPage.tsx lines 97-124
const fetchKpis = useCallback(async () => {
  if (!scorecardId) return;
  if (DEMO_MODE) {
    setKpis(DEMO_KPIS);
    setLoading(false);
    return;
  }
  setLoading(true);
  setError(null);
  try {
    const token = getAccessToken();
    const res = await fetch(`/api/v1/sdbip/scorecards/${scorecardId}/kpis`, {
      headers: { Authorization: `Bearer ${token}` },
    });
    // ...
  } catch {
    setKpis(DEMO_KPIS);
    setError(null);
  } finally {
    setLoading(false);
  }
}, [scorecardId, getAccessToken]);
```

**Key points:**
- `if (DEMO_MODE)` comes BEFORE `setLoading(true)` — prevents loading flash entirely
- `setLoading(false)` must be called explicitly in the demo branch (initial state is `true`)
- `return;` early — no API call is attempted

### Pattern 2: Demo Data Constants

**What:** Define a module-level constant `DEMO_*` array/object with realistic South African municipal data. Already established for IDP cycles, scorecards, and KPIs.

**For `IdpDetailPage`**, the demo data needs:
- `DEMO_CYCLE`: A single `IDPCycle` object (keyed by `cycleId`)
- `DEMO_GOALS`: Array of `IDPGoal` with national KPA values
- `DEMO_OBJECTIVES_BY_GOAL`: `Record<string, IDPObjective[]>` map

Since `IdpDetailPage` receives a `cycleId` URL param, and demo cycle IDs are `'demo-1'` and `'demo-2'` (from `IdpPage`'s `DEMO_CYCLES`), the demo data in `IdpDetailPage` should be keyed by cycle ID or provide a single default set that displays for any ID.

**Recommended approach:** A single default demo dataset that renders regardless of `cycleId` in demo mode — this is simpler and consistent with `SdbipKpiPage` which uses `DEMO_KPIS` regardless of `scorecardId`.

**For `ActualsPage`**, the demo data needs:
- `DEMO_KPI`: A single `SDBIPKpi` object
- `DEMO_ACTUALS`: Array of `SDBIPActual` with realistic quarterly data, traffic light statuses

**For `EvidencePage`**, the demo data needs:
- `DEMO_ACTUAL_INFO`: A single `ActualInfo` object
- `DEMO_DOCUMENTS`: Array of `EvidenceDocument` objects

### Pattern 3: Mutation Guard (lower priority)

For action handlers (`handleTransition`, `handleAddGoal`, `handleAddObjective`, `handleCreate`), the demo guard should be a no-op return at the function top — or optionally a console.warn. Do not show error state for attempted mutations in demo mode since they are user-initiated and can be silently ignored.

```typescript
const handleTransition = async (event: string) => {
  if (DEMO_MODE) return; // no-op in demo mode
  // ... real implementation
};
```

### Anti-Patterns to Avoid

- **Setting `loading: true` before the DEMO_MODE check:** This causes a one-render-frame flash of "Loading..." text. The guard must come BEFORE `setLoading(true)`.
- **Forgetting `setLoading(false)` in the demo branch:** Initial state is `loading: true`. If `DEMO_MODE` guard returns early without setting `setLoading(false)`, the page stays in the loading spinner state forever.
- **Using `catch` fallback to load demo data instead of an early return:** The catch fallback is a backup mechanism but introduces a network request round-trip. The `if (DEMO_MODE) { ...; return; }` early return prevents any network I/O entirely.
- **Conditional `useEffect` based on `DEMO_MODE`:** React hooks cannot be conditional. The guard goes inside the callback, not around the hook.

## Common Pitfalls

### Pitfall 1: `loading` State Initialized to `true`

**What goes wrong:** All three affected pages (`IdpDetailPage`, `ActualsPage`, `EvidencePage`) initialize `loading` state as `useState(true)`. If the `DEMO_MODE` guard returns early but forgets to call `setLoading(false)`, the page shows the loading text indefinitely.

**Why it happens:** The initial `true` is intentional — it prevents a flash of "empty state" before the first fetch completes. But it means the early-return path must explicitly set `false`.

**How to avoid:** Always call `setLoading(false)` in the `if (DEMO_MODE)` branch before `return`.

**Warning signs:** Page stuck on "Loading IDP cycle..." or "Loading actuals..." text in demo mode.

### Pitfall 2: Multi-Step Fetch in `IdpDetailPage.fetchAll`

**What goes wrong:** `fetchAll` in `IdpDetailPage` makes 3 sequential/parallel API calls: cycle, goals, and then N objectives fetches (one per goal). A naive demo guard that only skips the first call would still attempt the others.

**Why it happens:** The entire `fetchAll` function body needs to be bypassed, not just the first fetch.

**How to avoid:** Place the `if (DEMO_MODE)` guard at the very top of `fetchAll`, before any `setLoading(true)` and before any `fetch()` call. The demo branch sets all three state variables (`cycle`, `goals`, `objectivesByGoal`) and calls `setLoading(false)`.

**Warning signs:** Page briefly shows cycle header but then glitches on goals section.

### Pitfall 3: `cycleId` / `scorecardId` Param Mismatch in Demo Data

**What goes wrong:** If demo data is keyed by cycle ID (e.g., `'demo-1'`, `'demo-2'`) and the user navigates to `/pms/idp/demo-1`, the keyed lookup works. But if the user bookmarks a real ID like `/pms/idp/some-uuid`, no demo data is found and the page renders empty.

**Why it happens:** Detail pages receive IDs from URL params that may not match demo data keys.

**How to avoid:** Use a single default demo dataset that is returned regardless of the ID param. Follow the `SdbipKpiPage` pattern: always show `DEMO_KPIS` regardless of `scorecardId`.

**Warning signs:** Empty page in demo mode when navigating directly to a detail URL.

### Pitfall 4: `getAccessToken()` Call in Demo Mode

**What goes wrong:** Some pages call `getAccessToken()` inside the fetch function. In demo mode without a real Supabase session, this might return `null` or throw.

**Why it happens:** `getAccessToken()` relies on the Supabase session being present.

**How to avoid:** The `if (DEMO_MODE) { ...; return; }` guard before any `getAccessToken()` call entirely prevents this code path.

**Warning signs:** Console error about missing token in demo mode.

### Pitfall 5: EvidencePage Depth

**What goes wrong:** `EvidencePage` is two levels deep in the SDBIP drill-down (`SdbipPage` → `SdbipKpiPage` → `ActualsPage` → `EvidencePage`). The parent pages all have demo guards, but the evidence page fetches against `/api/v1/sdbip/actuals/:actualId` and `/api/v1/evidence/:actualId` — neither of which exists in demo mode.

**Why it happens:** EvidencePage was not included in the initial DEMO_MODE guard sweep.

**How to avoid:** Add demo guard + `DEMO_ACTUAL_INFO` and `DEMO_DOCUMENTS` constants to `EvidencePage`.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Try-fetch-then-catch-fallback | Explicit `if (DEMO_MODE) { return; }` before fetch | Phase 32-33 sweep | Eliminates network I/O, prevents loading flash entirely |
| No demo mode concept | `VITE_DEMO_MODE=true` + `demoMode.ts` | Phase 32+ | Compile-time guard, zero runtime cost |

**Deprecated/outdated:**
- The catch-fallback pattern (still present in some pages as a secondary safety net) is NOT a replacement for the early return guard. Both can coexist: the guard prevents the API call, the catch handles unexpected real-mode failures.

## Open Questions

1. **Should mutation handlers be guarded in demo mode?**
   - What we know: `handleCreate`, `handleAddGoal`, `handleAddObjective`, `handleTransition` in `IdpDetailPage` all hit the API. In demo mode they will fail silently (network error caught in catch block).
   - What's unclear: Whether the user experience of "create modal opens, form submits, nothing happens" is acceptable in demo, or whether a visible "demo mode — read only" message is needed.
   - Recommendation: Guard mutations with `if (DEMO_MODE) return;` (silent no-op). The forms already close on success, so a no-op on submit with no error message is clean enough for demo purposes. If a richer UX is desired, a console warning suffices.

2. **Does `EvidencePage` need demo data or just a guarded empty state?**
   - What we know: `EvidencePage` is reached from "Evidence" button in `ActualsPage`. In demo mode, no real actual ID exists.
   - What's unclear: Whether the demo should show real-looking evidence documents or just an empty state.
   - Recommendation: Provide `DEMO_DOCUMENTS` with 1-2 realistic filenames (e.g., `Q1_2025_Water_Loss_Report.pdf`, `Photo_Evidence_BSD001.jpg`) for a polished demo. An empty state would look broken.
