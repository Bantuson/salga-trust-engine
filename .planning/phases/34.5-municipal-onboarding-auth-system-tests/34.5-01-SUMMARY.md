# Phase 34.5 Plan 01: Auth Login Tests and OnboardingWizardPage Rewrite Summary

One-liner: Deleted pervasive graceful-skip onboarding specs, rewrote OnboardingWizardPage for 9-step v2 wizard with correct storage key and step indicator, created strict 7-test auth-login.spec.ts covering 5 roles with post-login dashboard heading verification.

## What Was Built

### Task 1: Delete stale specs and rewrite OnboardingWizardPage

**Deleted:**
- `e2e-tests/tests/municipal/onboarding.spec.ts` — 6 tests all with `test.skip()` graceful paths that silently passed on any backend failure
- `e2e-tests/tests/journeys/onboarding-journey.spec.ts` — 7 tests with pervasive `.catch(() => false)` chains that always returned truthy

**Rewritten: `e2e-tests/fixtures/page-objects/dashboard/OnboardingWizardPage.ts`**
- Storage key changed: `salga_onboarding_wizard_data` -> `salga_onboarding_wizard_v3` (matches frontend `OnboardingWizardPage.tsx`)
- Step definitions: `static readonly STEPS` (9-element `as const` tuple) and `static readonly STEP_LABELS` matching frontend
- Step detection: `stepIndicator` locator `span[hasText: /Step \d+ of 9:/i]` + `getCurrentStep()` parses step number from indicator text
- `getStepNumber(): Promise<number>` parses step number from indicator for direct numeric assertions
- `goto()` method: navigate -> clear storage key -> reload -> suppress GSAP -> verify step indicator visible
- GSAP suppression via `page.evaluate(() => gsap.globalTimeline.timeScale(100))`
- Wait times reduced from 1500-2000ms to 500ms (GSAP is suppressed)
- Removed stale methods: `fillProfileStep()`, `fillTeamStep()`, old step title locators
- No graceful-skip patterns — page object throws on timeout, spec decides how to handle

### Task 2: Create strict auth-login.spec.ts

**Created: `e2e-tests/tests/municipal/auth-login.spec.ts`**

7 tests:
1. Admin login -> asserts `h1/h2` with `/Municipal Operations Dashboard/i` visible
2. SALGA Admin login -> asserts `h1/h2` with `/SALGA Admin/i` visible
3. CFO login -> asserts `h1/h2` with `/CFO Dashboard/i` visible
4. Municipal Manager login -> asserts `h1/h2` with `/Municipal Manager Dashboard/i` visible
5. Citizen login (port 5174) -> asserts URL contains `/profile`
6. Invalid credentials -> asserts error element visible + URL still contains `/login`
7. Empty credentials -> asserts URL still contains `/login` (client-side validation holds)

Pattern:
- `browser` fixture (not pre-cached role pages)
- `context.close()` in `finally` block for reliable cleanup
- `suppressGsap(page)` called after `waitForURL`
- `test.slow()` at describe level
- Zero `test.skip()` calls (only appears in file header comment)
