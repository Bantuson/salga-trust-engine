## User Constraints (from CONTEXT.md)

### Locked Decisions

**Test scope & coverage**
- Full pipeline: Tests cover the entire SALGA-driven onboarding lifecycle — municipality submits access request, SALGA admin approves, invitation sent, user accepts via direct URL, admin completes onboarding wizard
- SALGA admin is the driver: Municipal onboarding is NOT self-service. SALGA admin handles approvals and triggers invitations. Municipal users receive email invitations to finalize role and account setup
- Invitation testing: Use direct invitation URL (bypass email delivery). No Supabase email hooks or test email providers
- Replace existing specs: Delete `onboarding.spec.ts` and `onboarding-journey.spec.ts`. Replace with comprehensive new test suite. Old specs have too many graceful-skip patterns that hide real issues

**Auth mode testing**
- Email+password only: Test the primary auth mode. Phone OTP and email OTP are deferred — too infrastructure-dependent for CI
- Key roles for login verification: Test login for admin, SALGA admin, CFO, municipal manager, citizen (5 roles). These cover the main dashboards and permission boundaries
- Verify post-login dashboard: After successful login, assert the user lands on their correct role-specific dashboard (e.g., admin → Municipal Operations Dashboard, CFO → CFO Dashboard)
- No registration tests: RegisterPage testing deferred. Users are onboarded via SALGA admin invitations, not self-registration

**Test reliability**
- Strict assertions: Tests must fail if expected state isn't met. No graceful-skip patterns for ambiguous state
- Reuse global test users: Use existing 20 profiles from global-setup (faster). No fresh-user-per-test isolation
- Disable GSAP animations: Set gsap.globalTimeline.timeScale(100) or CSS prefers-reduced-motion at test level. Eliminates 2-4s animation waits — faster and more reliable
- Live Supabase backend: Tests run against real Supabase instance (as configured in .env.test). Global-setup creates test data

**Negative & edge cases**
- Auth errors: Invalid credentials, wrong password, non-existent email. Assert error state appears (error element visible), not exact error message text
- Form validation — request-access: Missing required fields (municipality name, demarcation code, province, MM email), invalid email format, invalid demarcation code
- Form validation — onboarding wizard: Department names, SLA targets, ward count validation in wizard steps
- RBAC out of scope: Unauthorized route access already covered in authorization.spec.ts. This phase focuses on onboarding + auth flows only

## Summary

Phase 34.5 is a test replacement and expansion project. The two existing specs (`onboarding.spec.ts` and `onboarding-journey.spec.ts`) are pervasively graceful — every test has `test.skip()` paths for ambiguous state, meaning real regressions can silently pass. This phase deletes those specs and replaces them with strict, comprehensive tests covering: (1) the SALGA-driven access request → approval → invitation → wizard onboarding pipeline, and (2) login/auth flows for 5 key roles with post-login dashboard verification.

The codebase already has a mature E2E infrastructure: 20 pre-seeded test profiles, cached storageState auth fixtures, existing page objects for `LoginPage` and `OnboardingWizardPage`, and a journey spec pattern established in Phase 34-05. The `OnboardingWizardPage` (React) is a 9-step wizard. `AccessRequestsPage.tsx` is the SALGA admin view for reviewing/approving requests. `RequestAccessPage.tsx` is the public form.

The key research insight for planning: the existing `OnboardingWizardPage` page object was written for an old 6-step v1 wizard and its step titles are stale (it references "Profile", "Team", "Wards" steps which no longer exist in the v2 9-step wizard). The planner must account for updating or rewriting this page object to match the current step labels: `welcome`, `departments`, `invite-tier1`, `invite-directors`, `invite-dept-managers`, `create-teams`, `invite-supervisors`, `sla-config`, `pms-gate`.

**Primary recommendation:** Two spec files — `auth-login.spec.ts` (strict auth tests for 5 roles) and `onboarding-pipeline.spec.ts` (SALGA-driven pipeline from request-access through wizard completion) — plus an updated `OnboardingWizardPage` page object matching the current 9-step structure.

### Pattern 1: Strict Auth Test (email+password + post-login dashboard assertion)

What: Login with known credentials, assert navigation away from /login, assert role-specific dashboard element.
When to use: For all 5 role login tests.

```typescript
// Source: existing auth.ts fixture + LoginPage.ts page object
test('Admin login lands on Municipal Operations Dashboard', async ({ browser }) => {
  const context = await browser.newContext({ baseURL: 'http://localhost:5173' });
  const page = await context.newPage();
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.emailInput.waitFor({ state: 'visible', timeout: 15000 });
  await loginPage.emailInput.fill(admin.email);
  await loginPage.passwordInput.fill(admin.password);
  await loginPage.submitButton.click();
  // Strict: must navigate away from /login
  await page.waitForURL((url) => !url.pathname.endsWith('/login'), { timeout: 15000 });
  // Strict: role-specific heading must be visible
  await expect(page.locator('h1, h2').filter({ hasText: /Municipal Operations Dashboard/i }).first())
    .toBeVisible({ timeout: 15000 });
  await context.close();
});
```

### Pattern 2: GSAP Animation Suppression

What: Disable GSAP animations at test level to eliminate 2-4s wait overheads.
When to use: In beforeEach or immediately after page.goto() in every test.

```typescript
// Source: STATE.md Phase 34-05 decision + CONTEXT.md
await page.addInitScript(() => {
  // Disable GSAP animations globally
  if ((window as any).gsap) {
    (window as any).gsap.globalTimeline.timeScale(100);
  }
});
// OR inject via page.evaluate after navigation:
await page.evaluate(() => {
  if ((window as any).gsap) {
    (window as any).gsap.globalTimeline.timeScale(100);
  }
});
```

Note: `addInitScript` runs before page scripts load, so GSAP may not exist yet at injection time. Use `page.evaluate()` after `waitForLoadState('domcontentloaded')` instead, or inject a MutationObserver approach. The safest pattern is: navigate, wait for domcontentloaded, evaluate to speed up GSAP, then proceed.

### Anti-Patterns to Avoid

- **Graceful skip patterns:** `test.skip(true, 'X may not be available')` when the server is expected to be up. The replacement tests must not use this pattern — if a page doesn't load within timeout, that is a real failure.
- **Exact error text assertions:** `expect(errorMsg).toBe('Invalid login credentials')` — Supabase error messages change. Assert the error container is visible instead.
- **Stale OnboardingWizardPage selectors:** The existing page object references step titles from the old 6-step wizard ("Municipality Profile", "Invite Your Team", "Configure Your Wards", "Set SLA Targets"). These headings do NOT exist in the current `OnboardingWizardPage.tsx` v2. Any test using the old page object will silently fail the `isVisible()` check and return `false`.
- **Using old storage key:** `'salga_onboarding_wizard_data'` is in the old page object's `goto()`. The current wizard uses `'salga_onboarding_wizard_v3'`. Using the wrong key leaves old wizard state in localStorage, causing tests to resume mid-wizard unexpectedly.

## Common Pitfalls

### Pitfall 1: Stale OnboardingWizardPage Page Object
**What goes wrong:** Tests import `OnboardingWizardPage` and call `wizard.goto()`, which calls `localStorage.removeItem('salga_onboarding_wizard_data')`. The current wizard uses key `'salga_onboarding_wizard_v3'`. Old state persists and wizard resumes from a previous step instead of starting fresh.
**Why it happens:** Phase 34-03 rewrote the wizard but the page object was not updated.
**How to avoid:** Update `OnboardingWizardPage.ts` with the correct storage key AND update all step locators to match the 9-step v2 titles (step indicator text: "Step N of 9: {label}").
**Warning signs:** Wizard appears to start on step 2 or 3 unexpectedly; `getCurrentStep()` returns `'unknown'`.

### Pitfall 2: GSAP Animation Race Conditions Without Suppression
**What goes wrong:** `waitFor({ state: 'visible' })` times out on elements that are off-screen during GSAP entrance animations.
**Why it happens:** GSAP animates from `opacity: 0, y: 50` — the element exists in DOM but is not visible until animation completes.
**How to avoid:** Call `page.evaluate(() => { gsap.globalTimeline.timeScale(100); })` immediately after `waitForLoadState('domcontentloaded')` on every page navigation.
**Warning signs:** Tests pass locally (slow GSAP) but fail in CI (faster machine, different timing); `waitFor` timeouts on elements that visually appear.

### Pitfall 3: SALGA Admin Dashboard Prefill — Mock Data State
**What goes wrong:** The "Approve & Onboard" test navigates to `/access-requests` and finds no "pending" requests because a previous test already clicked "Reject" (optimistic state update persists in React state across tests using the same auth context).
**Why it happens:** `AccessRequestsPage` uses local React state for `requests` — clicking "Reject" in one test mutates state. If the page is not reloaded between tests, state persists.
**How to avoid:** Each test that needs a fresh `/access-requests` view must navigate to the page freshly (new `goto()` call). React state resets on navigation. Do NOT test approve + reject in the same page session without reloading.
**Warning signs:** "Approve & Onboard" button is missing when expected; no pending rows visible.

### Pitfall 4: Invitation URL — Supabase Session Conflict
**What goes wrong:** When testing invitation acceptance via direct URL, the test page has an active Supabase session for a different user (the inviting user). The invitation URL may conflict with the existing session or fail to recognize the invitation token.
**Why it happens:** Supabase stores session in localStorage. When a test navigates to an invitation URL in a context already holding a session, Supabase may redirect to the authenticated dashboard instead of the invitation acceptance flow.
**How to avoid:** Use a fresh browser context (no existing auth) when testing invitation URL acceptance. The existing `auth.ts` teardown closes contexts — ensure invitation tests use `browser.newContext()` without any pre-loaded storageState.
**Warning signs:** Navigation to invitation URL immediately redirects to `/` or `/onboarding` with the wrong user's state.

### Pitfall 5: RequestAccessPage Form — Updated Fields vs. Old Test
**What goes wrong:** `access-request.spec.ts` (the existing test) fills `contactName` and `contactPhone` fields that no longer exist in `RequestAccessPage.tsx` v2. The current form has `municipalManagerName` and `municipalManagerEmail` (not just `contactEmail`). Tests that reference the old `RequestAccessPage` page object will silently not fill the correct fields, then assert validation errors that may or may not appear.
**Why it happens:** The page object `RequestAccessPage.ts` was written for an older version of the form. The form was updated in Phase 34 to add demarcation code and MM email fields.
**How to avoid:** When writing new onboarding pipeline tests, directly use page locators based on the current form's label text (from `RequestAccessPage.tsx`): "Municipality Name", "Demarcation Code", "Province", "Municipality Category", "Municipal Manager Name", "Municipal Manager Email". Do not rely on the old page object's `fillForm()` method without verifying field names match.
**Warning signs:** Submitting the form shows no validation errors even when fields are empty (because the wrong fields were targeted).
