# Phase 34.2 Plan 03: Dashboard & Public Pages Select Migration Summary

Migrated all 9 page files in frontend-dashboard and 2 page files in frontend-public from native `<select>` elements to the unified glass-styled `Select` component from `@shared/components/ui/Select`.

## What Was Built

Completed the final wave of the dropdown menu styling migration. All native OS-chrome select dropdowns have been eliminated from both dashboards' pages directories.

**Task 1 — 5 dashboard pages:**
- **AccessRequestsPage**: 2 selects (status filter, province filter)
- **ActualsPage**: 1 select (quarter picker in submit form)
- **IdpDetailPage**: 1 select (national KPA in add goal form)
- **OnboardingWizardPage**: 2 selects (SDBIP level, department for team creation)
- **OversightDashboardPage**: 1 select (investigation reason in MPAC flag form)

**Task 2 — 6 remaining pages:**
- **PmsSetupWizardPage**: 5 selects (category, sdbip_layers with parseInt, scoring_method, director assignment, ticket category mapping) — highest complexity
- **RequestAccessPage**: 2 selects (province, municipality category)
- **RoleApprovalsPage**: 3 selects (role filter, municipality filter, status filter)
- **SdbipKpiPage**: 1 select (unit of measurement in new KPI form)
- **ProfilePage (frontend-public)**: 1 select (municipality picker, disabled state preserved)
- **ReportIssuePage (frontend-public)**: 1 select (category picker, GBV trigger logic preserved)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Removed unused `optStyle` constant from PmsSetupWizardPage**
- **Found during:** Task 2
- **Issue:** After migrating all 5 native selects in PmsSetupWizardPage, the `optStyle` constant was declared but never read. This would produce a TS6133 TypeScript error.
- **Fix:** Removed the `const optStyle = fieldStyles.selectOption as React.CSSProperties;` declaration and its comment block.
- **Files modified:** `frontend-dashboard/src/pages/PmsSetupWizardPage.tsx`
- **Commit:** 6c0bbfc

**2. [Rule 2 - Missing functionality] Updated `handleCategoryChange` signature in ReportIssuePage**
- **Found during:** Task 2
- **Issue:** The `handleCategoryChange` function in ReportIssuePage was typed as `(e: React.ChangeEvent<HTMLSelectElement>)` — incompatible with the unified Select's `onChange: (value: string) => void` API.
- **Fix:** Refactored to `(newCategory: string)` — the GBV trigger logic was preserved exactly (checks for 'GBV/Abuse', calls setShowGbvConsent, etc.), only the event signature changed.
- **Files modified:** `frontend-public/src/pages/ReportIssuePage.tsx`
- **Commit:** 6c0bbfc
