## Summary

Phase 34.2 is a focused UI consistency migration: replace all native `<select>` elements across two dashboards with a single unified glass-styled custom Select component. The groundwork is already solid вЂ” the project has two overlapping custom dropdown implementations (`shared/components/ui/Select.tsx` with full keyboard nav/ARIA, and `frontend-dashboard/src/components/settings/GlassSelect.tsx` with the target glass styling) that need to be merged into one canonical component.

The migration scope is concrete: 23 files in frontend-dashboard (excluding GlassSelect.tsx itself) and 2 files in frontend-public contain native `<select>` elements. An additional 2 files (`AuditLogSection.tsx`, `TeamDefaultsSection.tsx`) use GlassSelect and need updating to the shared import path. The `@shared` Vite alias is already configured in both dashboards and frontend-public already uses `@shared/components/ui/GlassCard`, `Button`, etc., so the import path `@shared/components/ui/Select` will resolve immediately without Vite config changes.

The key engineering challenge is the API difference between `<select>/<option>` and the custom `<Select options={[...]} />` pattern: every native select's `onChange={(e) => handler(e.target.value)}` becomes `onChange={(value) => handler(value)}`, and inline `<option>` children become an `options` prop array. One special case is RoleSwitcher, which currently sits inside the navbar and needs smart upward-flip logic when space is tight below.

**Primary recommendation:** Upgrade `shared/components/ui/Select.tsx` in place (adding `size` prop and fixing styling to GlassSelect aesthetic), then do a systematic file-by-file migration in batches grouped by domain (components first, then pages), ending with GlassSelect deletion and RoleSwitcher special treatment.

### Pattern 1: Unified Select Component API

**What:** The unified `Select` at `shared/components/ui/Select.tsx` accepts a `size` prop (`'sm' | 'md' | 'lg'`) controlling padding and font sizes. All other behavior (keyboard nav, click-outside, ARIA, styling) is identical across sizes.

**When to use:** Every place that currently has `<select>/<option>` JSX.

**Example вЂ” current native select pattern:**
```tsx
// BEFORE вЂ” native select
<select
  value={filters.status}
  onChange={(e) => setFilterStatus(e.target.value as StatusType)}
  style={{ padding: '0.5rem', background: 'var(--surface-elevated)', ... }}
>
  <option value="">All</option>
  <option value="open">Open</option>
  <option value="resolved">Resolved</option>
</select>
```

**Example вЂ” migrated unified Select:**
```tsx
// AFTER вЂ” unified Select
import { Select } from '@shared/components/ui/Select';

<Select
  value={filters.status}
  onChange={(value) => setFilterStatus(value as StatusType)}
  options={[
    { value: '', label: 'All' },
    { value: 'open', label: 'Open' },
    { value: 'resolved', label: 'Resolved' },
  ]}
  size="md"
/>
```

**Key API delta:** `onChange` receives `string` (not `React.ChangeEvent<HTMLSelectElement>`). Cast the value inside the handler where typed state is needed (e.g., `value as 'A' | 'B' | 'C'`).

### Pattern 2: GlassSelect Consumers Migration

**What:** `AuditLogSection.tsx` and `TeamDefaultsSection.tsx` currently import `GlassSelect` from `./GlassSelect`. They use `ariaLabel` and `maxWidth` props. The unified `Select` uses `label` (optional) and `fullWidth` (boolean). The `maxWidth` prop becomes a wrapper `div` `style` on the call site.

**Example вЂ” GlassSelect to unified Select migration:**
```tsx
// BEFORE
import { GlassSelect } from './GlassSelect';
<GlassSelect
  value={filterTable}
  onChange={setFilterTable}
  ariaLabel="Filter by table"
  options={[{ value: '', label: 'All tables' }, ...]}
/>

// AFTER
import { Select } from '@shared/components/ui/Select';
<Select
  value={filterTable}
  onChange={setFilterTable}
  options={[{ value: '', label: 'All tables' }, ...]}
  size="md"
  fullWidth={false}
/>
```

### Pattern 3: RoleSwitcher Smart Flip

**What:** The RoleSwitcher lives in the navbar header. Its dropdown must flip upward when insufficient space exists below. Detect available space via `getBoundingClientRect()` on the trigger element and compare to `window.innerHeight`.

**Example вЂ” position flip logic:**
```tsx
const [dropPosition, setDropPosition] = useState<'down' | 'up'>('down');

const handleOpen = () => {
  if (triggerRef.current) {
    const rect = triggerRef.current.getBoundingClientRect();
    const spaceBelow = window.innerHeight - rect.bottom;
    // Flip threshold: 240px (max-height of panel) + 8px buffer
    setDropPosition(spaceBelow < 248 ? 'up' : 'down');
  }
  setIsOpen(!isOpen);
};

// In dropdown panel style:
style={{
  position: 'absolute',
  top: dropPosition === 'down' ? 'calc(100% + 4px)' : undefined,
  bottom: dropPosition === 'up' ? 'calc(100% + 4px)' : undefined,
  // ...
}}
```

### Pattern 4: Size Preset Values

Based on existing usages and the GlassSelect trigger style (`padding: '0.5rem 0.75rem'`):

| Size | Trigger Padding | Font Size | Use Case |
|------|----------------|-----------|----------|
| `sm` | `6px 10px 6px 10px` | `var(--text-xs)` (0.75rem) | RoleSwitcher navbar, compact inline filters |
| `md` | `8px 2.5rem 8px 12px` | `var(--text-sm)` (0.875rem) | Standard form fields (default) |
| `lg` | `12px 2.5rem 12px 16px` | `var(--text-base)` (1rem) | Prominent filter bars, full-width form selects |

Note: GlassSelect used `0.5rem 0.75rem` вЂ” this maps cleanly to the `md` preset.

### Pattern 5: data-lenis-prevent on Scrollable Panel

The dropdown panel `div` (with `overflowY: auto` and `maxHeight: 240px`) must have `data-lenis-prevent` to prevent the Lenis smooth scroll library from intercepting mouse wheel events inside the dropdown. This is the canonical pattern used in NotificationBell, Sidebar, OrganogramTree, and all PMS modals.

```tsx
<div
  ref={listRef}
  role="listbox"
  data-lenis-prevent      // в†ђ required: prevents Lenis scroll capture
  style={{ maxHeight: '240px', overflowY: 'auto', ... }}
>
```

### Anti-Patterns to Avoid

- **Keeping GlassSelect as re-export alias:** Adds confusion; user decision is to delete it outright, no aliasing.
- **Using `e.target.value` in onChange:** The unified Select's `onChange` delivers `string` directly, not a DOM event. Any consumer that does `onChange={(e) => handler(e.target.value)}` will break вЂ” must become `onChange={(value) => handler(value)}`.
- **Applying `fullWidth` default incorrectly in navbar:** RoleSwitcher needs `fullWidth={false}` (it's inline in a flex row). Most form field selects want the default `fullWidth={true}`.
- **Gold chevron SVG from old Select.tsx:** The current shared `Select.tsx` uses a gold chevron (`stroke='%23ffd54f'`). Per locked decisions, the chevron color must be `var(--text-muted)`. Update the SVG data URI stroke color to `%23c7b3ba` (the hex value of `--text-muted: #c7b3ba`).
- **Box-shadow glow on open trigger:** Current `Select.tsx` applies `boxShadow: isOpen ? '0 0 0 2px rgba(0, 191, 165, 0.15)' : 'none'`. Per locked decisions, NO box-shadow glow вЂ” remove this.
- **Dark background on dropdown panel:** Current `Select.tsx` uses `rgba(30, 20, 40, 0.92)`. Per locked decisions, use GlassSelect's lighter `rgba(255, 255, 255, 0.15)`.

## Common Pitfalls

### Pitfall 1: onChange API Mismatch
**What goes wrong:** Migrating `<select onChange={(e) => setState(e.target.value as SomeType)}>` by just swapping the tag name. The unified Select calls `onChange(value: string)` вЂ” no event object.
**Why it happens:** React HTML `select` passes `ChangeEvent<HTMLSelectElement>`; custom component passes string directly.
**How to avoid:** In each file migration, update the handler signature: `(e) => handler(e.target.value)` в†’ `(value) => handler(value)`. For typed casts, move the cast inside: `(value) => handler(value as SomeType)`.
**Warning signs:** TypeScript will immediately error on `value.target` вЂ” treat TS errors as the migration checklist.

### Pitfall 2: Options with Dynamic Values (parseInt, etc.)
**What goes wrong:** `<select onChange={(e) => setLayers(parseInt(e.target.value, 10))}>` вЂ” numeric value stored in state. After migration to Select, `onChange` returns string and `parseInt` call must be preserved.
**Why it happens:** Select always delivers `string`; caller is responsible for type conversion.
**How to avoid:** `onChange={(value) => setLayers(parseInt(value, 10))}` вЂ” no change to conversion logic, only the signature.
**Warning signs:** PmsSetupWizardPage has `sdbip_layers: parseInt(e.target.value, 10)` вЂ” flag this during migration.

### Pitfall 3: Missing `data-lenis-prevent` on Scrollable Panel
**What goes wrong:** The dropdown panel scrolls fine with keyboard but mouse wheel on the panel scrolls the page instead, closing the dropdown.
**Why it happens:** Lenis intercepts all wheel events globally unless `data-lenis-prevent` is applied.
**How to avoid:** Add `data-lenis-prevent` to the dropdown panel `div` in the updated `shared/components/ui/Select.tsx`. Since this is in the shared component, all consumers get it for free.
**Warning signs:** Only noticeable at runtime; TypeScript won't catch it.

### Pitfall 4: RoleSwitcher Flip Logic вЂ” State Update Timing
**What goes wrong:** Computing flip direction inside `render` instead of on open click. If the component re-renders between trigger click and dropdown render, the position may flicker.
**Why it happens:** DOM measurements must happen at click time (synchronous with user intent).
**How to avoid:** Compute `dropPosition` in the click handler before `setIsOpen(true)`, storing it in local state. Not in `useEffect` or during render.

### Pitfall 5: GlassSelect `maxWidth` prop вЂ” Lost at Migration
**What goes wrong:** `TeamDefaultsSection.tsx` uses `<GlassSelect maxWidth="320px" ...>`. The unified Select has no `maxWidth` prop.
**Why it happens:** GlassSelect was a custom component with custom API.
**How to avoid:** Wrap the `<Select>` in a `<div style={{ maxWidth: '320px' }}>` at the call site. The Select's `fullWidth={true}` default will fill the wrapper.

### Pitfall 6: `ariaLabel` vs `label` prop naming
**What goes wrong:** GlassSelect uses `ariaLabel` prop (sets `aria-label` on the button). The unified Select uses `label` (renders a visible `<label>` element above the trigger).
**Why it happens:** Different component APIs.
**How to avoid:** For AuditLogSection filter dropdowns that use `ariaLabel` without a visible label, use `fullWidth={false}` and don't pass `label`. The button's `aria-haspopup` and `aria-expanded` provide sufficient accessibility context. If truly needed, add an optional `ariaLabel` prop to the unified Select that sets `aria-label` on the trigger button.

### Pitfall 7: Dropdown z-index Below Modal Overlay
**What goes wrong:** A Select inside a modal renders its dropdown behind the modal backdrop.
**Why it happens:** Modal overlays use high `z-index`. The dropdown uses `z-index: 50`.
**How to avoid:** The existing modals in this project (CreateIdpModal, CreatePaModal, etc.) use `z-index: 1000` for the overlay. The dropdown's `z-index: 50` works fine when the modal itself is a stacking context at `z-index: 1000` вЂ” the dropdown is still above the modal content inside the stacking context. This is the same pattern as GlassSelect inside settings modals, which already works.

## Open Questions

1. **`ariaLabel` prop in unified Select**
   - What we know: GlassSelect's `ariaLabel` prop sets `aria-label` on the trigger button for filter dropdowns without visible labels (e.g., AuditLogSection). Unified Select's `label` prop renders a visible `<label>` element вЂ” different purpose.
   - What's unclear: Whether to add `ariaLabel` prop to the unified Select for invisible-label use cases, or just accept that audit filter dropdowns will have `aria-haspopup` + `aria-expanded` for screen readers.
   - Recommendation: Add optional `ariaLabel?: string` prop to unified Select that sets `aria-label` on the trigger `<button>`. This is a small addition that preserves accessibility for the settings filter selects. The planner can decide at plan time.

2. **frontend-public ProfilePage + ReportIssuePage: CustomSelect vs @shared Select**
   - What we know: ProfilePage uses native `<select>` (municipality picker with `disabled` when not editing). ReportIssuePage uses native `<select>` (category picker). The `CustomSelect` component already exists in `frontend-public/src/components/CustomSelect.tsx` and has keyboard nav + glass styling.
   - What's unclear: Whether to migrate ProfilePage and ReportIssuePage to `@shared/components/ui/Select`, or replace with the local `CustomSelect`. The locked decision says "migrate to shared Select" вЂ” frontend-public's vite.config already has `@shared` alias pointing to `shared/`.
   - Recommendation: Use `@shared/components/ui/Select` for all two public files. The `@shared` alias is confirmed to resolve in frontend-public (GlassCard, Button, etc. already imported via `@shared`). The local `CustomSelect.tsx` is a redundant component and could be deprecated (but leaving it is not harmful вЂ” it has no native `<select>` inside it).

3. **RoleSwitcher auto-flip threshold**
   - What we know: The locked decision says "detect available space and flip dropdown upward if near bottom edge". The RoleSwitcher is in the 48px top header bar, meaning its dropdown would go down into the main content area вЂ” not near the bottom вЂ” so flip logic may rarely trigger for the RoleSwitcher itself.
   - What's unclear: Whether the flip logic should be built into the unified `Select` component (via `autoFlip?: boolean` prop) or only in RoleSwitcher. Given Claude's discretion on "flip detection threshold", making it a prop (`autoFlip={true}`) is recommended so any Select can opt in.
   - Recommendation: Add `autoFlip?: boolean` prop to unified Select. Default `false`. RoleSwitcher and any bottom-of-page selects can opt in. Implementation: `getBoundingClientRect()` on trigger ref in click handler, compare `window.innerHeight - rect.bottom` to panel maxHeight (248px with buffer).
