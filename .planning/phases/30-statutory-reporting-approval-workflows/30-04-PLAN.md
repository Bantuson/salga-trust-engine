```

POST   /api/v1/statutory-reports/{id}/transitions  -> StatutoryReportResponse
```

Create `frontend-dashboard/src/pages/StatutoryReportsPage.tsx`:

This is a single-page component with two sections: Report List (top) and Deadline Calendar (bottom).

**Component structure:**

```tsx
export function StatutoryReportsPage({ showForm, onCloseForm }: Props)
```

Props: `showForm` (boolean) and `onCloseForm` (callback) — same pattern as IdpPage.tsx and SdbipPage.tsx for consistency with PMS Hub "Create" button.

**State:**
- `reports: StatutoryReport[]` — fetched from GET /api/v1/statutory-reports/
- `deadlines: StatutoryDeadline[]` — fetched from GET /api/v1/statutory-reports/deadlines?financial_year={currentFY}
- `loading: boolean`
- `selectedFY: string` — default to current financial year (compute from current date, same FY logic as backend)
- `filterType: string | 'all'` — filter by report_type
- `generatingId: string | null` — report currently being generated
- `transitioningId: string | null` — report currently transitioning
- `error: string | null`

**API helper (inline or in a lib file):**
Use `fetch` with Authorization header from Supabase session token (same pattern as other pages). Base URL from env or relative path.

**Report List Section:**
- Header: "Statutory Reports" with filter controls (financial year input, report type dropdown)
- Table/cards showing each report:
  - Column: Type (human-readable: "Section 52 Q1", "Section 72 Mid-Year", "Section 46 Annual", "Section 121 Annual Report")
  - Column: Financial Year
  - Column: Status (badge with color):
    - drafting: gray badge
    - internal_review: blue badge
    - mm_approved: green badge
    - submitted: purple badge
    - tabled: dark green badge
  - Column: Generated (timestamp or "Not generated")
  - Column: Actions:
    - **Generate button**: visible when status is drafting or internal_review AND (pdf_storage_path is null OR user wants to regenerate). Calls POST /api/v1/statutory-reports/{id}/generate. Shows spinner while generating. On 202 response, show "Generation queued" toast.
    - **Transition buttons** (contextual based on current status and user role):
      - drafting: "Submit for Review" button
      - internal_review: "Approve" button (only for MM/CFO/admin roles)
      - mm_approved: "Submit to AG/Treasury" button (only for MM/admin)
      - submitted: "Table to Council" button (only for MM/Speaker/admin)
    - Each transition button calls POST /api/v1/statutory-reports/{id}/transitions with the appropriate event
    - **Download buttons**: visible when pdf_storage_path or docx_storage_path is not null
      - "PDF" button: links to GET /api/v1/statutory-reports/{id}/download/pdf
      - "DOCX" button: links to GET /api/v1/statutory-reports/{id}/download/docx

- Role-based button visibility: use `useAuth().getUserRole()` to check role. Define which transitions are available per role:
  ```typescript
  const TRANSITION_ROLES: Record<string, string[]> = {
    submit_for_review: ['pms_officer', 'department_manager', 'cfo', 'admin', 'salga_admin'],
    approve: ['municipal_manager', 'cfo', 'admin', 'salga_admin'],
    submit_external: ['municipal_manager', 'admin', 'salga_admin'],
    table: ['municipal_manager', 'speaker', 'admin', 'salga_admin'],
  };
  ```

**Create Report Form** (shown when showForm is true):
- Report type dropdown: Section 52, Section 72, Section 46, Section 121
- Financial year input (text, pattern: YYYY/YY)
- Quarter dropdown (Q1-Q4, only visible when type is Section 52)
- Title input (auto-generated default based on type + FY + quarter)
- Submit calls POST /api/v1/statutory-reports/

**Deadline Calendar Section:**
- Header: "Statutory Deadline Calendar — {selectedFY}"
- Grid/table showing the 7 deadlines:
  - Column: Report description
  - Column: Deadline date (formatted: "31 October 2025")
  - Column: Days remaining (computed client-side from today)
  - Column: Status indicator:
    - > 30 days away: green dot (safe)
    - 14-30 days: amber dot (approaching)
    - 7-14 days: orange dot (urgent)
    - 1-7 days: red dot (critical)
    - <= 0 (overdue): red pulsing dot + "OVERDUE" badge
  - Column: Auto-task created (checkmark if task_created=true)
- Sort by deadline_date ascending

**Styling:**
- Use CSS variables from design-tokens.css (NOT Tailwind — per Phase 27-03 decision)
- Use `@shared/components/ui/GlassCard` for containers
- Use `@shared/components/ui/Button` for action buttons
- Status badges use inline styles with background colors from CSS variables or literal color values matching the traffic-light convention
- Table uses standard HTML table with CSS grid or flex for responsive layout
